**3. Restructure FastAPI routing for /api prefix in backend/api/app.py:**

The frontend sends requests to `/api/...` (via API_BASE = '/api'). In dev, Vite proxy strips the /api prefix. In production (same container), there is no proxy -- FastAPI must handle the /api prefix itself.

Restructure `create_app()` to:
- Create a sub-application `api_app = FastAPI()` 
- Mount all existing routers (health, corpus, classify, viz) on `api_app` instead of `app`
- Mount `api_app` on the main `app` at path `/api`
- Add a health check at the ROOT level too (`/health`) for Railway's health check (Railway hits `/health` not `/api/health`)
- Add SPA fallback catch-all on `app` AFTER the /api mount

For SPA fallback, add a catch-all route on the main `app`:
```python
from fastapi.responses import FileResponse
from pathlib import Path

FRONTEND_DIR = Path(__file__).resolve().parents[2] / 'frontend' / 'dist'

@app.get('/{full_path:path}')
async def serve_spa(full_path: str):
    """Serve React SPA -- static files or index.html fallback."""
    file_path = FRONTEND_DIR / full_path
    if file_path.is_file() and FRONTEND_DIR in file_path.resolve().parents:
        return FileResponse(file_path)
    index = FRONTEND_DIR / 'index.html'
    if index.exists():
        return FileResponse(index)
    return FileResponse(status_code=404)
```

IMPORTANT: The catch-all MUST be registered AFTER the /api sub-app mount so /api routes take precedence. FastAPI handles WebSocket routes (`@router.websocket(...)`) separately from HTTP GET routes, so the catch-all will NOT intercept WebSocket upgrade requests.

IMPORTANT: Add path traversal protection -- verify `FRONTEND_DIR in file_path.resolve().parents` before serving to prevent directory traversal attacks.

The lifespan context manager stays on the MAIN app (not the sub-app) since it manages Redis connections used by both.

**2. Create entrypoint.sh:**
```bash
#!/bin/sh
set -e

# Start arq worker in background
echo "Starting arq worker..."
arq backend.worker.settings.WorkerSettings &
ARQ_PID=$!

# Start uvicorn in foreground
echo "Starting uvicorn on port ${PORT:-8000}..."
exec uvicorn backend.api.app:app --host 0.0.0.0 --port "${PORT:-8000}"
```

The `exec` replaces the shell with uvicorn so signals propagate correctly. The arq worker runs in the background. If uvicorn dies, the container exits and Railway restarts it (restart policy).

**3. Create Dockerfile (multi-stage):**

```dockerfile
# Stage 1: Build frontend
FROM node:20-slim AS frontend-builder
WORKDIR /app/frontend
COPY frontend/package.json frontend/package-lock.json ./
RUN npm ci --ignore-scripts
COPY frontend/ ./
RUN npm run build
# Produces /app/frontend/dist/

# Stage 2: Python runtime + models + cache
FROM python:3.12-slim

WORKDIR /app

# Install system deps
RUN apt-get update && apt-get install -y --no-install-recommends \
    curl \
    && rm -rf /var/lib/apt/lists/*

# Install Python deps first (layer caching)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY backend/ ./backend/
COPY config/ ./config/
COPY corpus/ ./corpus/
COPY scripts/utils.py ./scripts/utils.py

# Download models + cache from GitHub Release
# The RELEASE_URL build arg should point to a tarball containing:
#   data/models/  (runtime files only)
#   data/cache/   (pre-built cache)
ARG RELEASE_URL=""
RUN if [ -n "$RELEASE_URL" ]; then \
      echo "Downloading models + cache from $RELEASE_URL..." && \
      curl -fsSL "$RELEASE_URL" | tar xz -C /app; \
    else \
      echo "WARNING: No RELEASE_URL provided. Expecting data/models/ and data/cache/ in build context."; \
    fi

# Copy local data dirs as fallback (if they exist in build context and weren't downloaded)
COPY data/models/ ./data/models/
COPY data/cache/ ./data/cache/

# Copy built frontend
COPY --from=frontend-builder /app/frontend/dist ./frontend/dist

# Copy entrypoint
COPY entrypoint.sh .
RUN chmod +x entrypoint.sh

# Set Python path so backend imports work
ENV PYTHONPATH=/app

EXPOSE 8000

ENTRYPOINT ["./entrypoint.sh"]
```

IMPORTANT NOTES for the executor:
- The two-stage approach (not three) keeps it simple: Node builds frontend, Python runtime has everything else.
- `COPY data/models/ ./data/models/` will fail if data/models/ doesn't exist in the build context AND RELEASE_URL wasn't set. This is intentional -- the build should fail loudly if models are missing.
- For Railway deployment (Plan 02), the RELEASE_URL build arg will be set via railway.toml.
- For local docker compose, data/models/ and data/cache/ ARE in the build context (they exist locally), so no RELEASE_URL is needed.
- The `scripts/utils.py` COPY is needed because `backend/pipeline/precompute.py` and `backend/worker/settings.py` both import from `scripts.utils`.
- `PYTHONPATH=/app` ensures `backend.*` and `scripts.*` imports resolve correctly.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Internet -> Container | Public HTTP/WS traffic hits uvicorn directly (Railway TLS termination upstream) |
| Container -> Redis | arq worker + FastAPI both connect to Railway managed Redis via REDIS_URL |
| Docker build -> GitHub | Dockerfile downloads tarball from GitHub Release URL during build |
| SPA catch-all -> filesystem | Catch-all route serves files from frontend/dist/ directory |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-01 | Information Disclosure | Dockerfile layers | mitigate | No secrets in build args (RELEASE_URL is a public GitHub URL). .dockerignore excludes .env. REDIS_URL injected at runtime only. |
| T-5-02 | Information Disclosure | FastAPI docs endpoint | mitigate | Disable Swagger UI in production: set `docs_url=None, redoc_url=None` on the api_app sub-application. |
| T-5-03 | Tampering | SPA catch-all path traversal | mitigate | Validate `FRONTEND_DIR in file_path.resolve().parents` before serving any file. Reject paths that escape frontend/dist/. |
| T-5-04 | Denial of Service | Large file upload | accept | Existing 5MB limit in classify endpoint (implemented in Phase 2). No additional mitigation needed. |
| T-5-05 | Spoofing | GitHub Release tarball | accept | Tarball URL is set by user at build time (not dynamic). Risk is low for a personal project. |
| T-5-06 | Elevation of Privilege | Container escape | accept | Railway provides container isolation. No privileged mode requested. Standard Docker security posture. |
</threat_model>
