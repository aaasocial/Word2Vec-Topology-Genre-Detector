# Phase 08.1 — Context (Autonomous, Inline)

This sub-phase exists because Phase 8 Wave-1.5's full gid integrity audit (run during the
duplicate-gid patch) found 141 of 240 books in `corpus/books.yaml` have a `(title, author)`
that does NOT match the text Gutenberg actually serves at the listed `gid`. The audit log is
at `.planning/phases/08-corpus-expansion/wave-1-5-full-gid-audit.log`.

## Decisions (Locked autonomously)

**D-1: Authoritative metadata source = gutendex.com**
Gutendex (https://gutendex.com/) is the public, no-auth, JSON-over-HTTPS Gutenberg metadata
mirror. Standard endpoint: `https://gutendex.com/books/?search=<title>+<author>`. Returns
canonical title, authors[], gutenberg_id, and language for each match. This is the cleanest
way to map `(title, author)` → correct `gid` for the 141 broken rows.

**D-2: Mismatch classification = fuzzy-title + author-lastname**
- Fuzzy title match (rapidfuzz token_set_ratio ≥ 85) AND author last-name match → BENIGN
  (e.g., "Moby-Dick" vs "Moby Dick; Or, The Whale" by Melville).
- Either fails → SERIOUS (needs gid correction or substitution).
- Audit log's 141 entries are an upper bound; expected ~80–120 truly serious.

**D-3: Repair strategy = three-tier cascade per book**
1. Try gutendex lookup by `(title, author)`. If exactly-one English-text match → use that gid.
2. If multiple matches → pick the highest-downloaded match with matching author last name.
3. If zero matches OR matches fail integrity check → fall back to **pool substitution**:
   pick the next available candidate from `corpus_candidates.yaml`'s oversubscribed pool for
   that genre. The Wave-1 design includes a 2× pool exactly for this case.

**D-4: Entry gate for retrain = strict gid integrity audit returns 0 SERIOUS mismatches**
The audit script (existing: agent's classifier from Wave-1.5) must return 0 serious mismatches
across all 240 books before the Wave-2 precompute pipeline re-runs. Benign formatting
differences are allowed (logged, not blocking).

**D-5: Re-run scope = full Wave-2 pipeline (scripts 02 → 03 → 04 → 05 + precompute_all(15))**
Corpus changes invalidate every cache key (BUG-05 invariant). No partial re-run; full
deterministic regeneration. Lineage sidecar must rotate `corpus_hash`.

**D-6: Frozen hyperparameters = unchanged**
All 7 v1-frozen hyperparameters in `config/params.yaml` (window=15, k=200, α=0.7, etc.)
remain unchanged. This is a data refresh, not a model change.

**D-7: Per-genre book count target = 30 (same as Wave 1)**
Wave-1's deterministic generator emits exactly 30 per genre. If pool substitution depletes
a genre's pool below 30, that's a Phase-8.2 problem — flag and stop.

**D-8: Commit cadence = atomic per logical step**
- (a) candidates.yaml fix → commit
- (b) books.yaml regeneration + raw re-fetch → commit
- (c) entry-gate audit (must pass) → commit (gate result + log)
- (d) Wave-2 retrain artifacts → commit
- (e) SUMMARY + lineage diff → commit

**D-9: Failure mode = halt + handoff if gate fails**
If after best-effort repair the entry-gate audit still shows >5 serious mismatches, halt and
write a HANDOFF.md describing what remains. Don't ship a partially-fixed corpus.

**D-10: Out of scope (deferred)**
- Re-evaluating per-genre author-distribution balance (Phase 8 v2 corpus design)
- Adding new genres or books beyond the 240 slots
- Modifying any pipeline script logic (scripts 02-05 are frozen)
- Updating documentation outside this sub-phase (`v1_to_v2_migration.md` gets a patch trail
  appended — that's it)
