## Phase 8.1 Journey

### Session 1 (2026-05-25, 51 min) — Scaffolding + first halt

`scripts/audit_corpus_gids.py` was authored and the pre-repair audit was run against
all 240 books in `corpus/books.yaml`. Result: **240 rows / 95 BENIGN / 145 SERIOUS /
0 MISSING_GUTENDEX**.

`scripts/repair_corpus.py` was authored with gutendex bulk-by-author lookups (4-worker
parallel, 6-retry exponential backoff, per-worker `requests.Session()`, 90s timeout).
First execution stalled out on gutendex.com's slow response times (P50 30-45s, P95
60-90s, intermittent HTTP 429). The session hit budget and **halted-and-handed-off**
per plan D-9 + threat model T-08.1-01.

**Commits:** `bf75706`, `57ed151`, `d90ab4b`, `e1a99e5`

### Session 2 (2026-05-26, 132 min) — Bug fixes + repair runs + gate fail

The next session resumed from the HANDOFF and discovered two latent bugs in
`scripts/repair_corpus.py`:

1. **Title-score-first sort missing** (commit `6d83899`): `find_best_match_in_cache()`
   was sorting candidates by `download_count` first, which caused multiple distinct
   titles (e.g., `Return of Tarzan` and `Beasts of Tarzan`) to collapse onto the
   single most-downloaded gid (`Tarzan of the Apes`). Fix: sort by
   `(-title_score, -download_count)` and seed `excluded_gids` with the running set of
   chosen new_gids.

2. **Audiobook-record rejection** (commit `6082c27`): some gutendex records have only
   `<gid>-readme.txt` (a sentinel pointing at an mp3 download) as their `text/plain`
   URL, no actual prose. The original heuristic `is_audiobook_record()` was too lax.
   Fix: a record is audiobook-only iff **all** text/plain URLs end in `-readme.txt`.

The second repair run (`514ed2e`) produced **100 GUTENDEX_LOOKUP gid-changed edits +
3 BENIGN_CONFIRMED + 42 NEEDS_SUBSTITUTION**. Then `scripts/build_corpus.py`
regenerated `corpus/books.yaml` with 240 unique gids. A cross-genre gid collision at
gid 4276 (E.M. Forster's "A Room with a View" vs Gaskell's "North and South") was
patched inline. Post-audit: **154 BENIGN / 86 SERIOUS / 0 MISSING** — 40.7% reduction
but still failing D-9's `≤5 SERIOUS` gate. **Halt** with HANDOFF rev-2.

**Commits:** `6d83899`, `bb98e8d`, `6082c27`, `514ed2e`, `b5ecbcf`, `97d3ebd`

## Deviations from Plan

### Scope expansion: drop strategy added (Rule 4 escalation → user-authorized)

The original 08.1-01-PLAN.md envisioned a repair-only resolution path (T-1 audit → T-2
repair → T-3 regenerate → T-4 retrain → T-5 docs) with D-9 specifying halt-and-HANDOFF
if `>5 SERIOUS` remained. Sessions 1+2 reduced 145 → 86 SERIOUS but could not clear
the gate. The drop strategy was a new resolution path authorized by the user, not in
the original plan. Documented as **T-FIN-1 (drop) + T-FIN-2 (retrain) + T-FIN-3 (docs)**
in this completion summary.

### Rule-3 (blocking input) auto-fixes

**1. [Rule 3 — Blocking] gutendex.com rate-limiting**
- 90s timeout, 6-retry exponential backoff, per-worker session, 4-worker parallelism.
- Documented in earlier sessions' SUMMARY revisions.

**2. [Rule 3 — Blocking] Stale pipeline artifacts (Session 4)**
- `data/raw/` had 300 .txt files (300-154=146 stale from prior sessions).
- `data/processed/` had 274 .json files (stale).
- `data/features/` had ~2400 files (stale window-15 entries for dropped books).
- **Fix:** `scripts/clean_stale_pipeline_artifacts.py` reads books.yaml as ground truth
  and removes only files whose gid is NOT in the manifest (146 raw, 123 processed,
  752 features removed). Preserves idempotent retrain semantics.

### Rule-1 (Bug fix) auto-fixes (Session 2)

**3. [Rule 1 — Bug] Title-score-first sort missing in repair_corpus.py**
- See "Session 2" above. Commit `6d83899`.

**4. [Rule 1 — Bug] Audiobook-record sentinels not detected**
- See "Session 2" above. Commit `6082c27`.

### Per-genre count assertion: no patches needed

The original plan's threat model T-08.1-03 (pool depletion) was anticipated as a
risk if repair_corpus.py couldn't find substitutes. The drop strategy resolved this
by accepting variable per-genre counts (15-25) instead of forcing 30/genre. No
script-level assertions on `>=30/genre` exist outside `scripts/test_build_corpus.py`
(tests, not pipeline scripts), so no patches were required.

## Deferred to v2.1 (Phase 8.2 candidate)

**Re-source the 86 dropped entries.** Use authoritative author bibliographies
(Wikipedia author pages, Library of Congress catalog entries, canonical book lists
like Bloom's Western Canon for the literary/historical genres) to map each dropped
(title, author) to its correct Gutenberg gid via gutendex search by canonical name.
Target restoration: v2.1 corpus = 240+ books with all gids verified-correct, then
re-trigger the retrain pipeline (which would again rotate corpus_hash + w2v_sha256
per BUG-05).

The infrastructure exists in this codebase:

- `scripts/audit_corpus_gids.py` — strict gid integrity auditor
- `scripts/repair_corpus.py` — gutendex bulk-by-author repair
- `scripts/drop_serious_rows.py` — drop executor (idempotent)
- `scripts/build_corpus.py` — books.yaml regenerator
- `scripts/01_download_corpus.py` → `precompute.py` — retrain chain

The gap is the (title, author) → correct_gid mapping for the 86 entries. The
author-bulk-fetch approach used in Session 2 failed for these because either (a) the
canonical author name on gutendex differs from the candidates.yaml form (accents,
initials), or (b) the correct gid is buried under high-download_count works by the
same author that mislead the title-score sort.

A v2.1 manual pass with `?search=<title>+<lastname>` per individual row would
likely resolve 60-70 of the 86 (the others are genuinely missing from
Project Gutenberg with that title-author combination).
