## Decisions Made

1. **Halt-and-HANDOFF per plan's D-9.** gutendex.com rate-limit + long Wave-2 retrain
   pushed the wall-clock budget to 2-3 hours. Per the plan's threat model T-08.1-01
   ("gutendex API unavailable / rate-limit") + D-9 ("halt if entry gate audit still
   shows >5 serious mismatches"), the right move is to commit verified-good work,
   author HANDOFF.md, and signal to the orchestrator that a fresh session is needed.

2. **Author-bulk-fetch architecture.** 145 SERIOUS rows → 54 distinct lastnames.
   One `?search=<lastname>&languages=en` query (paginated) returns up to 160 hits
   per author. Local matching does the rest. This is ~3x query reduction over the
   naive `?search=<title> <lastname>` per-row approach.

3. **Idempotency over partial-state recovery.** Repair script writes
   `corpus_candidates.yaml` only at end-of-match. Aborted partial runs leave the
   file untouched. Trade-off: every retry starts the gutendex fetch from scratch.
   This was the right call because: (a) Wave 1's `corpus_candidates.yaml` is the
   only source-of-truth we have for the (title, author) → expected-gid mapping,
   and (b) partial overwrites would corrupt the file in failure modes.

4. **Uniform `rapidfuzz.token_set_ratio >= 85` + lastname-match for audit.**
   Wave-1.5's audit used different fuzzy logic and reported 141 SERIOUS; ours
   reports 145. The 4-row difference is benign — entries like "Mrs Dalloway" vs
   "Mrs. Dalloway" are flagged SERIOUS by the strict threshold but will land in
   BENIGN_CONFIRMED after repair (when gutendex agrees the gid is fine).

### Script-level minor deviations

**2. [Rule 3 — Blocking] gutendex search 90s timeout, 6 retries**
- **Found during:** First repair run (workers=1, 30s timeout) was timing out P95.
- **Issue:** 30s default Python requests timeout < gutendex's actual response time.
- **Fix:** Bumped to 90s timeout. Added 6-retry exponential backoff with INITIAL_BACKOFF=4s
  (handles gutendex's ~few-second rate-limit window better than the 1s initial that
  Python's requests defaults assume).
- **Commit:** `57ed151`

**3. [Rule 3 — Blocking] requests.Session not thread-safe for in-flight ops**
- **Found during:** First parallel-worker run (8 workers, all sharing one session) hit
  ConnectionPool warnings + RemoteDisconnected errors mid-batch.
- **Issue:** Shared session with parallel workers caused HTTP keep-alive + reuse contention.
- **Fix:** Each worker gets its own `requests.Session()` (closed at end via try/finally).
- **Commit:** `57ed151`

**4. [Rule 1 — Bug] 8 parallel workers hammered gutendex into immediate 429**
- **Found during:** Second repair run (8 workers, parallel).
- **Issue:** 8 simultaneous queries to gutendex.com triggered immediate rate-limit on the
  very first batch.
- **Fix:** Dropped to 4 workers (`--workers 4` default). Combined with the 6-retry
  backoff this is the right concurrency level — every worker gets through within
  the 60-128s backoff window without overwhelming the API.
- **Commit:** `57ed151` (workers default codified in script)

## Issues Encountered

### gutendex.com rate-limiting / slowness (the central blocker)

**Symptoms (observed across 4 separate runs over ~50 min):**
- P50 response time: 30-45s.
- P95: 60-90s (worse than my initial 20s timeout assumption).
- Status 429 returns when concurrency >4.
- Intermittent `RemoteDisconnected('Remote end closed connection without response')`.

**Mitigation in repair_corpus.py:**
- 90s timeout per request.
- 6 retries with exponential backoff (4, 8, 16, 32, 64, 128s).
- Per-worker `requests.Session()`.
- 4-worker parallelism (sweet spot between throughput and 429-floor).
- Direct gutenberg.org HTML fallback per-gid for rows the author-cache misses
  (e.g., accent issues like 'Brontë' vs the audit's 'bronte').

**Actual observed steady-state throughput tonight:** ~1.4 authors/minute,
projecting 38-40 min total fetch time for 54 lastnames.

### Accented-name search miss (handled by fallback)

**Symptom:** `?search=bronte` returns 4 hits (incidental, e.g., "James Bronterre O'Brien"),
none matching the canonical Brontës. The canonical entries in gutendex use "Brontë" with
diaeresis. My lastname normalisation strips diacritics, so the search misses.

**Mitigation:** The repair script falls back to per-gid HTML probe of
`https://www.gutenberg.org/ebooks/<gid>` — that page's `<meta name="author">` tag
contains the canonical name with diacritic. If it matches the expected `(title,
lastname)` of a SERIOUS row, we mark the row BENIGN_CONFIRMED (the audit was just
being too strict on accented characters).

**Caveat:** Some Brontë rows may still be marked NEEDS_SUBSTITUTION if the audit-flagged
gid genuinely is wrong (e.g., Wave 1.5 already moved Jane Eyre to gid 1260 which is
actually Charlotte Brontë's Jane Eyre — that row should be BENIGN_CONFIRMED). The
HTML probe catches this.
