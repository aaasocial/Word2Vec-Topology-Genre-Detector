# Plan 08.1-01 — Corpus Integrity Rebuild

## Objective

Rebuild `corpus_candidates.yaml` against Gutenberg's authoritative metadata (via gutendex.com)
for the rows flagged as serious mismatches in `wave-1-5-full-gid-audit.log`, regenerate
`corpus/books.yaml`, re-fetch raw text files, and re-run the full Wave-2 precompute pipeline.
Entry gate before retrain: strict gid integrity audit returns 0 serious mismatches.

## Why

`08-02-SUMMARY.md` documents the 6 duplicate-gid defects we already patched (Wave 1.5).
`wave-1-5-full-gid-audit.log` enumerates 141 single-mapping mismatches not yet fixed. The
current Phase-8 v2 SVM is trained on a corpus where ~58% of books bind to the wrong text.
Phase 8 Waves 3 + 4 (validation report + v2.0 GitHub Release) cannot credibly proceed on
this state. This plan delivers a clean v2 corpus + retrained model that Waves 3 + 4 can use.

## Tasks

### T-1: Build the mismatch classifier and pre-repair audit

Write `scripts/audit_corpus_gids.py`:
- For each book in `corpus/books.yaml`:
  1. Fetch the gutendex metadata for that `gutenberg_id` via
     `https://gutendex.com/books/?ids=<gid>` (single request can pass `?ids=82,121,...` to
     batch up to 32 ids per call — use batching).
  2. Compute `rapidfuzz.fuzz.token_set_ratio(yaml_title, gutendex_title)` and
     `author_lastname_match(yaml_author, gutendex_authors)`.
  3. Classify: BENIGN (both pass), SERIOUS (either fails), MISSING_GUTENDEX (gid not
     resolvable — unlikely, but possible if Gutenberg withdrew the work).
- Emit a structured report (per-book row: gid, genre, yaml_title, yaml_author,
  gutendex_title, gutendex_author, classification, reason).
- Write to `.planning/phases/08.1-.../08.1-gid-audit-pre.log`.

**Verification:** Script exits 0, report has 240 rows. The "SERIOUS" count establishes the
true scope (expected somewhere in [80, 120]).

**Commit:** `feat(08.1-01): scripts/audit_corpus_gids.py — strict gid integrity auditor`

### T-2: Repair the SERIOUS mismatches (gutendex lookup + pool substitution)

Write `scripts/repair_corpus.py`:
- Read the SERIOUS rows from `08.1-gid-audit-pre.log`.
- For each: query gutendex by `(title, author)`:
  - `https://gutendex.com/books/?search=<urlencode(title + " " + author_lastname)>`
  - Filter results to English, sort by `download_count` desc, pick first exact-author-lastname
    match.
  - If exactly one good candidate → use that gid.
  - If multiple ambiguous → log decision rationale, pick highest download_count.
  - If zero → mark row as `NEEDS_SUBSTITUTION`.
- For `NEEDS_SUBSTITUTION` rows: read `corpus_candidates.yaml` pool for that genre, pick the
  next un-used candidate (not already in books.yaml), and verify ITS gid maps to expected
  title/author via the gutendex API. If that also fails, escalate to next pool entry.
- Write a structured repair log (per row: old_gid, new_gid, action [gutendex_match|pool_sub],
  rationale) to `.planning/phases/08.1-.../08.1-repair-decisions.log`.
- Update `corpus_candidates.yaml` in place with the corrected `gutenberg_id` values.

**Verification:** No SERIOUS row left without a resolved new_gid. Repair log explains every
change.

**Commit:** `fix(08.1-01): repair {N} corpus rows — gutendex lookups + pool substitutions`

### T-3: Regenerate books.yaml + re-fetch raw files + post-repair audit

- Re-run `scripts/build_corpus.py` against the corrected `corpus_candidates.yaml`. This is
  idempotent and will:
  - Re-fetch any raw `data/raw/<gid>.txt` whose gid changed
  - Re-emit `corpus/books.yaml` with the corrected entries
  - Re-emit `.planning/research/v2/corpus_build.log`
- Re-run the audit (T-1's script) → write to `08.1-gid-audit-post.log`.

**Entry gate:** `08.1-gid-audit-post.log` must report 0 SERIOUS mismatches. If >5 remain,
halt and write `HANDOFF.md` describing what's left. If 1–5 remain, log and decide per-case
in the repair decisions log.

**Commit:** `feat(08.1-01): regenerate books.yaml with corrected gids — gate pass`

## Threat model (brief)

- **T-08.1-01 — gutendex API unavailable / rate-limit:** mitigated by batching `?ids=` queries
  (32 per request) and sleeping 0.5s between bulk calls. Fallback to pool substitution if
  the API is fully down.
- **T-08.1-02 — Gutenberg title formatting drift:** mitigated by fuzzy match (token_set_ratio
  ≥ 85). Edge cases (e.g., series titles) logged in repair decisions.
- **T-08.1-03 — Pool depletion:** Wave-1 pool was 2× per genre (60 candidates → 30 books).
  If repairs deplete a pool, halt and HANDOFF.
- **T-08.1-04 — Lineage corruption / partial retrain:** mitigated by full deterministic
  re-run (scripts 02–05 + precompute) + lineage sidecar verification.
